# Ресёрч: «фазы проекта» как фича gcode (project knowledge) 📖

> Вопрос Газика: фазы нужны не только для стройки gcode — они нужны ЛЮБОМУ проекту (новому, старому, даже не настроенному под ИИ), чтобы открыть «главу core» и понять, как всё работает. Как это делают топы, что показал наш собственный опыт, и куда это встроить.

## Как это решают топы

**DeepWiki / Devin Wiki (Cognition)** — ровно кейс «старый проект»: автогенерация wiki из репозитория — архитектура, ключевые модули, диаграммы связей, прозаические объяснения «как куски стыкуются». Devin переиндексирует репо и держит wiki актуальной. Вывод: анализ существующего кода → структурированные «главы» — рабочая, проверенная идея.

**Spec-driven инструменты**: Spec Kit (GitHub) — greenfield-first: /specify → /plan → /tasks, человек гейтит каждую стадию (наш процесс фаз — это оно). OpenSpec — brownfield-first, и его главный урок: **не документируй весь легаси разом** — пиши спеки только на вносимое изменение, покрытие растёт инкрементально.

**Главная боль индустрии — drift («Spec Debt»)**: доки/спеки не самообновляются; после пары месяцев они противоречат коду и путают и человека, и ИИ. Ни один инструмент это до конца не решил (Devin — переиндексацией, spec-тулзы — «резинкуй руками»).

## Мета-анализ: как мы строили gcode в этом чате
**Что сработало (забираем в продукт):**
- Фаза = один документ ≤5 минут: Зачем / Что получишь / Как / Шаги-чекбоксы / Решения.
- **Человеческий гейт**: фаза не стартует без «го» — «не наугад».
- Решения фиксируются В доке фазы (история «почему так» не теряется).
- Чекбоксы обновляются по ходу → фаза = живой статус, не мёртвый план.
- Ресёрч-доки как отдельный вид (research/*.md) перед крупными решениями.
- Витрина/скриншот-гейт для визуального.

**Что я упустил (урок):** design-system и ui-inventory возникли только по твоим комментам — их не было в моём делении на фазы. Значит нужен **каталог типовых фаз/глав**, чтобы ничего не пропускать: Research → Vision → Architecture/Concurrency → Design System (если есть UI) → UI Inventory → Core → … → Distribution. gcode должен подсказывать состав, а не полагаться на память оркестратора.

## Предложение: фича «Project knowledge» в gcode

**Два вида документов** (наш же опыт, обобщённый):
1. **Строительные фазы** (roadmap во времени) — для новых проектов и для крупных изменений в старых.
2. **Описательные главы** (структура: «как устроен core», «как работает биллинг») — для любого проекта; это ответ на «открыл и понял».

**Три сценария:**
- **Новый проект**: мастер — ИИ предлагает деление на фазы из каталога типовых (по типу проекта: библиотека/приложение/сервис), человек правит и гейтит. Наш процесс, автоматизированный.
- **Существующий (invictus)**: «Проанализировать проект» — агент строит overview + главы по подсистемам (DeepWiki-паттерн), но **инкрементально** (урок OpenSpec): сначала карта-обзор, главы наполняются по мере работы в этих зонах, а не «всё разом».
- **Проект без ИИ-настройки**: тот же анализ генерирует и CLAUDE.md/AGENTS.md скелет — gcode делает проект «ИИ-готовым» одной кнопкой. Да, эту функциональность даём: она дешёвая (это тот же анализ) и решает холодный старт.

**Анти-drift (наш ответ на главную боль — из наших же принципов «факты, не проза»):**
- Глава декларирует свои источники (файлы/папки). Задача смёржена и тронула эти файлы → глава автоматически помечается **stale** (бейдж в UI) — это ops-из-фактов для документации.
- Правило агенту в задаче: «изменил поведение подсистемы — обнови её главу» (как PROGRESS.md сейчас).
- Завершение задачи показывает: «затронуты главы X, Y — обновить?» (кнопка, агент обновляет).

**Куда в план: новая фаза 6 «Project knowledge»**, остальные сдвигаются (группы→7, ops→8, дистрибуция→9, remote→10). Почему сюда: требует только агентов (ф2) и UI (ф3); для invictus-режима это ценность №1; stale-механика дозреет вместе с git-ops (ф5). Полный документ фазы напишу на твоё чтение перед её стартом, по нашему правилу.
