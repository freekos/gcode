# UI-инвентарь gcode — что показываем, где, и почему 📖

> Правило, из которого всё выведено: **каждая зона отвечает на один вопрос пользователя.** Если элемент не помогает ответить на вопрос своей зоны — ему тут не место. Это спека для фазы 3b: из неё видно, какие компоненты нужны и что они обязаны уметь.

## Зоны и их вопросы

### 1. Сайдбар — «что у меня есть и что требует меня?» (v2 по фидбеку)
| Показываем | Зачем |
|---|---|
| **Дерево: ВСЕ проекты IDE → их задачи** (+ группы появятся в фазе 7) | индустрийный паттерн (Orca/ZCode/Conductor — у всех проект→задачи); всё хозяйство видно сразу |
| Задачи отсортированы ВНУТРИ проекта: needs_input → review → running → остальные, БЕЗ заголовков-групп | приоритет внимания сохранён, шум заголовков убран — статус несут точки-индикаторы |
| Статус-точка, имя задачи | статус читается до текста |
| Хоткей ⌘1-9 — ТОЛЬКО при наведении на карточку | тултип-политика: подсказка по запросу, не мебель |
| Бейдж вопроса на needs_input | вопрос агента виден без открытия задачи |
**Действия:** выбрать задачу (клик / ⌘1-9; клик задаёт и проект-контекст для ⌘N), ⌘N создать, свернуть сайдбар.
**Чего тут НЕТ и почему:** диффстатов и веток — это ответ «что изменилось», он живёт в контекст-панели.

### 2. Тред (центр) — «что агент делает/сделал и что ему сказать?»
| Показываем | Зачем |
|---|---|
| Шапка: имя, статус-бейдж, ветка (моно) | ориентация: где я и в каком состоянии задача |
| Сообщения диалога; юзер-сообщения выделены фоном | сканируемость диалога (паттерн Conductor) |
| Тул-события одной строкой: `[Edit] auth.ts +4 −1` | видно ЧТО агент трогал, без простыней JSON (паттерн ZCode) |
| Финальный структурированный отчёт (Summary / по репо) | «анти-хаос»: понять итог за 10 секунд (паттерн vibe-kanban) |
| Список тредов задачи (слева от треда, узкий) | вернуться к прошлым разговорам |
**Действия:** написать (⏎), очередь при работающем агенте, ⏸ стоп, переключить тред, клик по тул-событию → дифф файла (фаза 4).

### 3. Контекст-панель (справа) — «что изменилось и где агент по плану?»
| Показываем | Зачем |
|---|---|
| Per-repo блоки: имя (моно), ✎ N файлов, +A −D — **только тронутые репо**, остальные свёрнуты одной строкой «ещё N не тронуты» | наш мультирепо-уникум без шума: 12 worktree ≠ 12 блоков |
| Goal / Progress — чеклист из PROGRESS.md | живой план агента; файл в задаче → любой движок продолжит |
| Готовность: «чисто / есть незакоммиченное / готово к PR» | мостик к ops: понятно, можно ли двигать дальше |
**Действия:** клик по репо → дифф (ф4), git-действия (ф5).

### 4. Композер (низ треда) — «сказать агенту»
| Показываем | Зачем |
|---|---|
| Поле ввода + очередь-пометка при работающем агенте | сообщение не теряется и не блокирует |
| ◆ Движок ▾ + лимиты «↯ 58% 5h · 41% wk» | лимит принадлежит движку треда (твоё решение) |
**Действия:** ⏎ отправить/в очередь, ⏸ стоп, сменить движок.

### 5. Ops-модалка ⌘O — «что готово к следующему шагу, что где стоит, что было?»
Три зоны в одном окне (без табов): Задачи-pipeline с причиной-фактом и одной кнопкой следующего шага → Окружения компактно → История с фильтрами. Работающие задачи приглушены.

### 6. Создание ⌘N — «запустить работу за 5 секунд»
Глобальная модалка: ТОЛЬКО промпт («что сделать») + проект. Имя придумает ИИ. **Выбора репо нет** — worktree создаются на все репо проекта (CoW делает это дёшево), панель задачи показывает только тронутые; выбор остаётся флагом в CLI для редких случаев.

### 7. Палитра ⌘K — «сделать что угодно, не зная где оно»
Команды и переходы с хоткеями в строках (как Linear) — заодно **обучает хоткеям** ненавязчиво. Анти-туман: всё, что есть в приложении, находится здесь по названию.

### 8. Ассистент ⌘J (после MVP) — «объясни/найди/что я делал вчера»
Чат со снапшотом состояния + журналом ядра + докой продукта.

### 9. Статусбар — «всё ли живо?»
Проект · число задач · ● N агентов работают · версия. Тихий, моно, без кнопок.

## Политика тултипов (анализ: NN/g, Microsoft HIG, Material)
Индустрийный консенсус: тултипы — **экономно**, и никогда не повторять надпись элемента.
1. **Кнопка с текстом** → тултип НЕ нужен. Если у неё есть хоткей — тултип показывает **только хоткей** (задержка ~500ms).
2. **Icon-only кнопка** → тултип ОБЯЗАТЕЛЕН: «Действие · хоткей» (у нас icon-only минимум: свернуть панель, стоп, фильтры).
3. **Факты-сокращения** (✎3, +61 −3, «↯ 58% 5h») → тултип-расшифровка: «3 изменённых файла в server», «лимит 5-часового окна Claude».
4. Никаких тултипов-туров и обучалок: хоткеям учат палитра ⌘K (хоткей в каждой строке) и пустые состояния.
5. Доступность: тултип появляется и по клавиатурному фокусу, не только hover.
Итого: тултип — это расшифровка сжатого, а не дубль очевидного.

## Поведение модалок: overlay, Esc и черновики (анализ топов)
- **Клик по overlay закрывает ЛЮБУЮ модалку** (как палитры Linear/Raycast) — «залипшие» окна раздражают. Esc — всегда.
- **Ввод никогда не теряется:** у модалок с полями (создание задачи, ассистент) черновик персистентен — случайно закрыл → открыл → текст на месте (паттерн Notion/Linear). Именно черновик, а не запрет закрытия.
- **Опасные подтверждения** (Release, снос worktree с правками) — исключение: закрываются только явной кнопкой или Esc; клик мимо не считается решением.

## Настройки — отдельная страница, НЕ модалка (и вот почему)
Анализ: мультисекционные настройки в модалке — антипаттерн (табы в модалках работают плохо — Smashing/NN-выводы); Linear открывает настройки полноэкранной страницей (⌘,), Zed хранит всё в `settings.json` + UI поверх, Raycast — отдельное окно.
**Наш принцип глобальных модалок не ломается:** модалки — для ЧАСТЫХ лёгких действий поверх контекста (создать задачу, глянуть ops). Настройки — редкие и многосекционные → им нужна страница.

**Решение:**
- `⌘,` → страница настроек с секциями слева (паттерн Linear): **General** (проекты, пути), **Engines** (движки: бинарь, дефолтный, permission-режим), **Git** (шаблон ветки, конвенция коммит-сообщений — явно и видимо, не как в Cursor), **Appearance** (акцент + тихий/яркий, плотность), **Keybindings**, **Advanced** (state db, журнал, диагностика).
- **Источник правды — файл `~/.gcode/settings.json`** (паттерн Zed): UI редактирует файл; файл можно бэкапить/синкать/править руками. Соответствует нашей философии «факты в файлах».
- В настройках НЕ живёт операционное (это ops) и контекстное (это панель задачи). Настройки = то, что меняешь раз в месяц.
- В 3b строим минимум: Appearance + Engines; остальные секции — по мере появления функций.
