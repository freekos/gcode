# Дизайн-система gcode 📖

> Что это: единый источник правды о том, как выглядит и ощущается gcode — **токены** (именованные значения цветов/отступов/шрифтов/таймингов), **компоненты** (кнопки, поля, бейджи — собранные один раз) и **правила** их использования. Строится ДО первого экрана: дальше UI растёт настройкой, не переделкой.

## Как это делают топы (ресёрч 2026-07-04)

**Linear** — эталон:
- Цвета в **LCH** (пространство, равномерное для глаза: два цвета одной «светлоты» реально выглядят одинаково светлыми) — темы генерируются, а не рисуются руками: **вся тема из 3 переменных** (базовый цвет, акцент, контраст) вместо 98 захардкоженных.
- **4 уровня поверхностей** («элевации»): фон → панели/карточки → вложенное+hover → оверлеи (модалки/тултипы). Каждый уровень чуть светлее в тёмной теме — так строится глубина без теней.
- **Dark-first**: всё спроектировано на тёмной поверхности, светлая — производная. (Ровно наш случай.)

**Индустрийный стандарт токенов — 3 слоя:**
1. **Primitive** — сырая палитра без смысла: `gray-900`, `amber-500`. Меняется только при ребрендинге.
2. **Semantic** — смысл: `surface-1`, `text-muted`, `accent`, `status-running`. Ссылаются на primitives. **Здесь живёт темизация**: светлая тема = переприсвоение semantic-токенов, компоненты не трогаются.
3. **Component** — по надобности: `button-bg`. Ссылаются только на semantic.
Команды, пропускающие semantic-слой «для скорости», потом неделями выковыривают захардкоженные цвета.

**Механика 2026:** Tailwind v4 — конфиг в CSS (`@theme`), токены = CSS-переменные, один источник правды для утилит и рантайм-темизации.

## Наши решения

### Токены
- **3 слоя** как у всех; хранение — один файл `app/src/lib/design/tokens.css` (появится первым коммитом 3b).
- Цвета в **OKLCH** (наследник LCH, нативен в CSS): палитра генерится из 3 значений по рецепту Linear — база (глубокий графит), акцент, контраст.
- **4 поверхности**: `surface-0` (фон) → `surface-1` (сайдбар/панели) → `surface-2` (карточки, hover) → `surface-3` (модалки/поповеры).
- **Цвет = сигнал** (решение 3a): статусы `status-new/running/needs-input/review/done`, дифф `diff-add/remove`, один `accent`. Всё остальное — оттенки серого.
- Текст: `text-primary / secondary / muted / disabled`.

### Типографика («ZCode-баланс», утверждено)
- UI — **Inter** (variable), код/ветки/слаги/команды — **JetBrains Mono**.
- Компактная шкала: 12 / 13 (базовый) / 14 / 16 / 20. Плотность Linear, не лендинг.

### Геометрия и движение
- Отступы — сетка **4px** (4/8/12/16/24/32); радиусы 4/6/10; бордеры 1px `border-subtle`.
- Motion-токены: `fast 120ms` / `base 160ms`, ease-out, только opacity/transform. Пульс running-статуса — 2s мягкий.

### Компоненты
- **Headless-подход** (индустрийный стандарт для Svelte — bits-ui, аналог Radix): поведение/доступность из библиотеки, весь внешний вид — наш, на токенах. Никаких готовых UI-китов с чужим лицом.
- Стартовый набор 3b: Button, Input, Badge(статусы), Card(задача), Modal, Tooltip, Kbd(хоткей), Spinner/Pulse.
- **Витрина** — страница `/styleguide` внутри приложения со всеми токенами и компонентами: на ней делаем скриншот-сверку и утверждаем вид ДО сборки экранов.

## Где это в плане
Первый шаг фазы 3b: «Дизайн-система: tokens.css + базовые компоненты + витрина» → скриншоты тебе → «нравится» → только потом экраны. Так «нормальный UI» гарантирован конструкцией, а не надеждой.

## Вопрос к тебе
Акцентный цвет gcode (единственный яркий не-статусный цвет: выделение, фокус, активные элементы). Варианты: холодный синий (Linear/Zed-вайб) / фиолетовый / янтарный / зелёный / свой. Мой дефолт — холодный синий; финально покажу варианты на витрине вживую.
