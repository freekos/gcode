# v1 — что это, для кого и что must-have 📖

> Продуктовый анализ перед финишной прямой. Читается за 5 минут. В конце — 4 вопроса, они определяют порядок работ.

## Продукт одной строкой

**gcode — приложение для macOS, где ты запускаешь несколько ИИ-задач параллельно, каждая в своих изолированных git-worktree (сразу на все репо проекта), и спокойно ревьюишь их диффы — агент физически не может трогать git.**

## Кто пользователь v1

Разработчик вроде тебя: работает соло или ведёт направление, держит 1–10 репо в одной папке, уже пользуется Claude Code в терминале и упирается в его главный предел — **один чат = одна очередь**. Он хочет: поставил 3 задачи → пьёт кофе → вернулся → отревьюил диффы → отправил комментарии → смёржил. С телефона — позже (фаза 10), v1 — за маком.

Что он получает, чего нет у терминала и у конкурентов (vibe-kanban, Conductor, Crystal):
1. **Мультирепо-задача** — worktree на все репо разом, знание через корневой CLAUDE.md (наш уникум).
2. **Агент без git-прав** (deny-enforcement, проверен под bypass) — «агент не запушит и не намёржит».
3. **Ревью-петля**: дифф → комментарии пачкой → одним сообщением агенту → он чинит.
4. Статусы из фактов процессов, не из прозы агента.

## Путь пользователя (как этим пользуются)

**Первые 10 минут** (сейчас дыра — см. gap G1):
скачал .dmg → открыл → gcode сам проверяет: git есть? `claude` CLI есть и залогинен? → «+ Добавить проект» → указал папку с репо → «Что делаем?» → написал первую задачу → смотрит, как агент работает, имя и ветка «догоняют» → дифф → готово.

**Рабочий день** (уже работает):
утром 2–3 задачи подряд (мгновенное optimistic-создание) → работает своей работой, поглядывая на статус-точки → зашёл в задачу: карточка «Изменил файлы» → дифф-вкладка → выделил строки → комментарии ⏎ в пачку → «Отправить всё агенту» → агент чинит → **[конец v1-цикла: Git-ops]** «Создать MR» / «смёржить в ветку» кнопкой с подтверждением → архив задачи.

**Хвост цикла сегодня обрывается**: изменения остаются в worktree, и юзер должен идти в терминал. Это единственное место, где v1-петля не замкнута.

## Что уже есть (фазы 0–4.5) — ядро готово

Задачи+worktrees+deny-git+архив с патчем; живые треды (стрим, стоп, resume, очередь, ИИ-имена); вкладочный центр (тред/дифф/файл/md); дифф-ревью с пачкой комментариев; cowork-навигатор (прогресс/изменения-с-файлами/контекст); ⌘K/⌘P; вложения ⌘L и [+]; 74+ тестов ядра, CI, release.yml (.dmg draft). Дизайн-канвас (.pen) = источник правды UI.

## Что сознательно НЕ в v1

Project knowledge (фаза 6 — твой azi-кейс, едет в v1.1), группы/integration-ветки (7), ops-деплои (8), Telegram/аккаунты (10), проектный режим-чат, Codex-движок (селектор уже есть — «скоро»), Windows/Linux (msi собирается, но не тестим), поиск по тредам, edit-сообщения.

**Логика реза**: v1 продаёт одну петлю — «параллельные задачи → ревью → MR» — идеально. Всё, что не в петле, ждёт.

## Предлагаемый порядок до v1

1. **Смоук 4.5 на azi** (ты) → закрываем фазу.
2. **Фаза 5 Git-ops** (G2) — самая большая и самая ценная.
3. **G1+G3** онбординг + permissions (вместе — «чужой человек ставит и доверяет»).
4. **G5+G6** настройки + устойчивость.
5. **G4** EN-локаль.
6. **G7** дистрибуция → **v1 release** (тег, Homebrew, README, Show HN?).
