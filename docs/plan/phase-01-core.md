# Фаза 1 — Ядро: домен, состояние, worktrees ✅

## Зачем
Всё, что будет в gcode (агенты, UI, диффы, ops), стоит на трёх китах: **модель данных** (что такое проект/задача/группа), **состояние** (где это хранится и почему не ломается) и **provisioning** (как задача получает свои worktree, не убивая машину). Если это сделать небрежно — гонки и потери данных будут преследовать нас вечно. Поэтому фаза 1 — самая строгая: минимум магии, максимум тестов.

## Что получишь в конце (пощупать)
Из терминала:
```bash
gcode project add ~/Codebase/work/azi          # регистрируешь проект (папка с репо внутри)
gcode task new azi "починить логин" --repos azi-server,azi-crm
# → создана папка задачи: worktree на КАЖДЫЙ репо от нужной ветки,
#   .env скопирован, зависимости готовы (CoW-клон), контекст-файл в корне
gcode task ls azi                              # список задач со статусами
gcode task archive t1 && gcode task restore t1 # архив (worktrees снесены) и восстановление
gcode journal --today                          # что происходило сегодня (журнал операций)
```
И всё это — с полным набором тестов (включая конкурентные: 10 задач создаются параллельно — ничего не рвётся).

## Как будем делать (простым языком)

### 1. Доменная модель — четыре сущности
- **Project** — папка, внутри которой живут твои git-репозитории (как `~/Codebase/work/azi`). gcode сканирует её и запоминает реестр репо.
- **Task** — единица работы. Главная особенность: **мультирепо** — при создании выбираешь, какие репо затронуты, и задача получает **git worktree на каждый** под общим корнем: `<project>/.gcode/tasks/<task>/<repo>/`. *(Worktree — это «вторая рабочая копия» репозитория: файлы свои, история общая — диск не дублируется.)* Задача без группы — просто задача проекта.
- **Group** — необязательная папка-ярлык для задач (визуальная группировка) + опционально integration-ветка. В этой фазе — только membership (метки), ветки групп придут в фазе 6.
- **Thread** — беседа агента внутри задачи. В этой фазе только заготовка типа (сами агенты — фаза 2).

### 2. Состояние — SQLite вместо JSON-файла
Опыт cc показал: один большой `state.json` рвётся при параллельной записи и требует самодельных бэкапов. Берём **SQLite** — встроенная база в один файл:
- **WAL-режим** *(журнал упреждающей записи: читатели не блокируют писателя, файл не бьётся при падении)*;
- каждая операция — транзакция: или применилась целиком, или нет;
- журнал операций (audit) — обычная таблица: «что я делал вчера» для будущего ассистента бесплатно.

### 3. Конкурентность — «без гонок by construction»
Три правила, которые делают гонки невозможными архитектурно, а не удачей:
- **Один писатель состояния.** Все изменения идут через одну очередь (актор): два изменения физически не могут случиться одновременно. *(Актор — цикл, который берёт команды из очереди по одной.)*
- **Git-операции сериализуются по репозиторию.** На один репо — одна git-команда за раз; разные репо — параллельно. Rust-компилятор сам следит, чтобы данные не расшарились между потоками без защиты.
- **Тяжёлое — в очередь.** Клоны зависимостей/инсталлы не бегут пачкой — максимум N одновременно.

### 4. Provisioning — «не убивать машину»
Создание задачи для каждого репо: `git fetch origin <ветка>` (только нужное) → `git worktree add` → копия `.env*` (кроме `.env.example`) → зависимости каскадом: **pnpm** → install через общий стор (почти бесплатно); иначе **CoW-клон** `node_modules` *(мгновенная копия на APFS/btrfs, занимает ~0 байт, пока файлы не меняются)*; полный `npm install` — никогда молча. Архив задачи = снести worktrees, состояние остаётся, restore возвращает. Свой хук на нестандартный сетап.

### 5. Тесты — три уровня, с первого дня
- **unit** — логика домена, все ветвления (занятые имена, грязные репо, архивные сущности);
- **e2e** — на настоящих временных git-репо (мини-полирепо создаётся в tmp на каждый прогон);
- **конкурентные** — параллельное создание/архив/запись, ловим гонки до пользователей.

## Решения (зафиксированы 2026-07-03)
1. **Папка задач: `<project>/.gcode/tasks/<task>/…`** (скрытая — не мозолит глаза в проводнике/редакторе).
2. **Ветка задачи создаётся сразу** при создании задачи, в каждом worktree; имя генерится из названия.
3. **Архив с незакоммиченными правками: предупредить + авто-сохранить патч.** Патч — текстовый файл-дифф со всеми правками (`git diff`), лежит в данных задачи; при restore предлагается приложить обратно (`git apply`) — правки не теряются.
